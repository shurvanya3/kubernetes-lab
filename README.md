# Kubernetes: деплой веб-приложения

Данный проект представляет собой лабораторную работу по DevOps. Цель проекта — продемонстрировать полный цикл упаковки и развертывания веб-приложения: от написания Dockerfile до настройки отказоустойчивой архитектуры и маршрутизации в локальном кластере Kubernetes.

## Описание проекта

Приложение представляет собой простой HTTP-сервер на **Python (Flask)**. Оно возвращает JSON с информацией о себе: название сервиса, имя хоста (внутри контейнера/пода) и текущую версию приложения. 

В проекте реализовано два варианта развертывания:
1. Легковесный запуск через **Docker Compose** с использованием Nginx в качестве reverse-proxy.
2. Полноценное развертывание в **Kubernetes (Minikube)** с балансировкой нагрузки, управлением конфигурацией и доменной маршрутизацией.

### Стек технологий
* **Код:** Python 3.12, Flask
* **Контейнеризация:** Docker, Docker Compose
* **Проксирование:** Nginx
* **Оркестрация:** Kubernetes, Minikube
* **K8s Объекты:** Deployment, Service, ConfigMap, Ingress

## Инструкция по запуску

### Пререквизиты
Убедитесь, что у вас установлены: `docker`, `docker-compose` (или `docker compose`), `minikube` и `kubectl`.

---

### Вариант 1: Запуск через Docker Compose

В этом варианте локально поднимаются два контейнера: само приложение и Nginx, который принимает трафик на порту 80 и проксирует его на приложение. Прямой доступ к приложению закрыт.

1. Клонируйте репозиторий и перейдите в папку проекта.
2. Запустите сборку и старт контейнеров:
   ```bash
   docker compose up -d --build
   ```
3. Проверьте работоспособность приложения:
```bash
curl http://localhost
curl http://localhost/health
```
4. Для остановки проекта:
```bash
docker compose down
```
### Вариант 2: Запуск в Kubernetes (Minikube)
В этом варианте приложение деплоится в локальный кластер в виде нескольких реплик (подов). Трафик балансируется между ними через Ingress-контроллер по доменному имени `devops-lab.local`.
1. Запустите кластер и включите Ingress:
```bash
minikube start
minikube addons enable ingress
```
2. Соберите Docker-обаз:
Переключите контекст Docker на внутренний демон Minikube, чтобы кластер определил образ, запустите сборку:
```bash
eval $(minikube docker-env)
docker builtd -t devops-lab:v2 ./app
```
3. Примените манифесты Kubernetes:
```bash
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/deployment.yaml
```
4. Настройте файл `hosts`:
Чтобы Ingress смог маршрутизировать трафик по доменному имени, добавьте запись в файл hosts
- Определите IP-адрес кластера: `minikube ip`
- Откройте файл `/etc/hosts` (для Linux) или `C:\Windows\System32\drivers\etc\hosts` (для Windows) от имени администратора и добавьте строку:
```
<MINIKUBE_IP> devops-lab.local
```
5. Проверьте результат:
Выполните несколько запросов. Вы увидите, что поле `hostname` в ответе меняется - это демонстрирует балансировку нагрузки между подами
```bash
curl http://devops-lab.local
```
## Реализованные бонусные задания
В рамках проекта были успешно выполнены дополнительные задачи:

1. ConfigMap: Версия приложения (APP_VERSION) вынесена из манифеста Deployment в отдельный объект ConfigMap для удобного управления конфигурацией.
2. Ingress: Вместо NodePort настроен Nginx Ingress Controller. Приложение доступно не по IP и случайному порту, а по красивому локальному домену devops-lab.local.
3. Rolling Update: Протестировано обновление приложения без простоя (zero-downtime deployment). При изменении версии Kubernetes плавно заменяет старые поды на новые. Также проверен механизм отката изменений (kubectl rollout undo).
